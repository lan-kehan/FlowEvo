## Objective

## AutoTTS Analogy

Think in the same terms as AutoTTS probing:
- width: how many particles/branches to create;
- depth: how far each branch is advanced before observing or committing;
- probe: `preview`, the only semantic observation action;
- branch/refine: `backward` from preview anchors;
- prune: remove active particles that no longer deserve compute;
- answer: finish using the best available preview/active trajectory.

Early previews can be noisy. Late previews are expensive but more reliable.
Backward creates new candidates around an anchor; it is useful only when the
anchor selection, target time, noise policy, and confirmation budget are tied to
observed evidence.

## Novelty Requirement

This is mandatory. A valid proposal must be behaviorally non-trivial relative to
the best recent controller and to the seed baselines.

The following do not count as meaningful proposals by themselves:
- comment-only edits;
- renaming variables;
- changing one threshold by a tiny amount;
- reordering the same fixed stages without changing decisions;
- copying the previous best controller with only cosmetic cleanup.

If you build on a prior strategy, add a clear new mechanism. Examples of valid
mechanisms include:
- adaptive branch allocation from score gaps or uncertainty;
- explicit weak-budget repair, such as protecting beta=0.25 from losing to ODE;
- SDE scouts only where stochastic forward motion has a bounded role;
- ODE/euler fallback when preview evidence is weak;
- two-stage preview confirmation before backward refinement;
- score-gap based pruning and late confirmation;
- beta-dependent noise policy that changes exploration radius and child count;
- budget-aware stopping that spends only when the expected best-score gap merits it.

Pick one or two mechanisms. Keep the policy simple enough to generalize.

## Beta Schedule Discipline

Implement a clear helper such as `_schedule(beta) -> dict`.
Every behavior-critical hyperparameter must be a deterministic function of beta.
Keep the schedule smooth or monotonic wherever it affects budget use.

Recommended shape:
- beta=0: deterministic ODE/euler anchor behavior, near 10 NFE;
- low beta: very limited extra previews; must not waste enough compute to lose
  badly to the nearest ODE baseline;
- mid beta: adaptive scouts plus selective refinement;
- high beta: accuracy-first, more conservative pruning, enough diversity and
  confirmation to spend near the budget ceiling when useful.

Avoid many brittle breakpoints and magic constants. A small number of named
parameters is better than a large table.

## Implementation Expectations

Design a controller that responds to observations:
- preview before expensive branching when evidence is needed;
- compare ranked preview scores, score gaps, uncertainty, time, and active
  status through public state records;
- allocate children to the most promising anchors rather than uniformly;
- prune clear losers after observing them;
- preserve diversity at high beta so the controller does not collapse onto the
  first good preview;
- stop early only when the budget level and preview evidence justify it.

Use `backward` as anchor-based re-noising/refinement, not uncontrolled noise
spam. If using `solver="sde"`, bound it carefully and explain where it helps.
